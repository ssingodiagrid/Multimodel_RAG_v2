# Phase 6: End-to-End Multimodal RAG with a ColPali-like Approach

This phase pushes the project beyond text-plus-caption retrieval into a more native **visual document understanding** pipeline. Instead of relying primarily on extracted text, table summaries, or image captions, we will treat document pages themselves as multimodal objects and retrieve the most relevant **visual regions / patches** directly.

The goal is to build a ColPali-inspired retrieval system that can reason over layout, typography, tables, charts, and surrounding visual structure together, while still integrating with the existing Streamlit app, Qdrant storage, and Gemini-based generation pipeline.

---

## 1. Why Phase 6 Is Needed

Phase 5 improves performance on tables and charts by converting them into text-like artifacts:
- Tables become structured summaries plus raw table text
- Images become captions generated by a vision model

This works well for many queries, but it still loses some information:
- Fine-grained layout relationships between text, labels, and numeric values
- Visual evidence spread across multiple nearby regions on a page
- Charts and scanned content where OCR/captioning may omit important details
- Complex page structures where meaning depends on spatial arrangement

Phase 6 addresses this by retrieving directly from **page images and visual patches**, not only from text extracted from them.

---

## 2. Core Objective

Implement an end-to-end multimodal RAG pipeline inspired by **ColPali / ColBERT-style late interaction retrieval**, where:
- document pages are rendered as images,
- pages are segmented into smaller visual patches,
- each patch receives a multimodal embedding,
- user queries are embedded into the same retrieval space,
- retrieval uses a late-interaction / MaxSim-style scoring strategy,
- the top visual regions or full pages are passed to a multimodal LLM for final answer generation.

---

## 3. High-Level Architecture

### Retrieval Philosophy
- **Phase 5:** Retrieve text chunks, table summaries, and image captions
- **Phase 6:** Retrieve directly from page images / visual patches using multimodal embeddings

### Proposed System Flow
1. Ingest PDF pages as rendered images
2. Segment pages into visual patches or layout regions
3. Generate multimodal embeddings for those patches
4. Store embeddings and metadata in Qdrant
5. Convert user query into a compatible query embedding
6. Score query against page patches using a late-interaction / MaxSim method
7. Return top patches and/or full source pages
8. Send retrieved visual context to Gemini or another multimodal LLM
9. Generate answer with visual evidence and patch/page attribution

---

## 4. Model and Tooling Strategy

### Candidate Models / References
- **ColPali / ColQwen-style approaches**
  - Best conceptual reference for visual patch retrieval with late interaction
- **PaliGemma**
  - Good candidate for multimodal page/patch representation experiments
- **Qwen2-VL / Qwen2.5-VL**
  - Strong modern VLMs for page understanding and visual QA
- **LLaVA / similar open VLMs**
  - Useful for experimentation and local prototyping
- **Gemini 2.0 Flash / Gemini multimodal models**
  - Strong option for answer generation over retrieved images/pages

### Supporting Tools
- **PyMuPDF (`fitz`)**
  - Render PDF pages to high-resolution images
- **Pillow / OpenCV**
  - Patch extraction, resizing, cropping, coordinate handling
- **Qdrant**
  - Store patch embeddings, metadata, and page references
- **LangChain / custom retrieval layer**
  - Integrate multimodal retrieval into the app pipeline

## 5. Phase 6.1: Visual Document Ingestion and Preprocessing

The first step is to transform each PDF page into a visual representation suitable for multimodal indexing.

### 1. Page Rendering
- Render each PDF page to a high-resolution image using `fitz`
- Store page-level metadata:
  - `source`
  - `page_number`
  - `width`
  - `height`
  - `render_dpi`

### 2. Patch / Region Segmentation
- Divide each page into smaller visual units
- Two possible strategies:
  - **Fixed grid patches**
    - Example: `4x4`, `6x6`, or `8x8` grid per page
  - **Layout-aware segmentation**
    - Detect text blocks, tables, figures, charts, and headings

### 3. Patch Metadata
- Every patch should preserve:
  - `page_number`
  - `patch_id`
  - `bbox` or coordinates `(x1, y1, x2, y2)`
  - `patch_type` if known (`text_region`, `table_region`, `chart_region`, `mixed`)
  - link to the full page image

### Deliverables
- Create a new parser / preprocessing module for page rendering and patch extraction
- Persist rendered pages and optionally cropped patch images under `data/processed/`

---

## 6. Phase 6.2: Multimodal Embedding Generation

This stage creates the retrieval representations used by the vector database.

### 1. Embedding Objective
- Generate embeddings that capture:
  - visible text
  - tables and chart structure
  - typography and layout
  - local visual context

### 2. Patch Embedding Pipeline
- For each patch:
  - load patch image
  - preprocess to target model input size
  - encode patch using the chosen multimodal embedding model
  - store embedding with patch metadata in Qdrant

### 3. Optional Multi-Vector Storage
- If the selected architecture supports token-level or patch-level embeddings, store:
  - one vector per patch, or
  - multiple vectors per patch/page for late interaction

### 4. Storage Design
- Store in a dedicated Qdrant collection such as:
  - `ifc_visual_patches`
- Metadata should include:
  - `source`
  - `page_number`
  - `patch_id`
  - `bbox`
  - `page_image_path`
  - `patch_image_path`
  - `content_type="visual_patch"`

### Deliverables
- New embedding module under `src/embeddings/`
- New ingestion/indexing pipeline integrated into `ingest.py` or a separate Phase 6 ingestion script

---

## 7. Phase 6.3: Multimodal Retrieval with Late Interaction

This is the core retrieval innovation of Phase 6.

### 1. Query Embedding
- Convert user text queries into multimodal-compatible query embeddings
- Ensure the query representation lives in the same embedding space as the visual patches

### 2. Retrieval Strategy
- Retrieve candidate patches from Qdrant
- Apply **late interaction** or **MaxSim-style** scoring:
  - compare query representation to patch representations
  - aggregate maximum similarities across patch/token dimensions
  - rank pages or patch sets by total evidence score

### 3. Patch-to-Page Aggregation
- Since answers are often page-level or multi-region:
  - aggregate top patch scores by page
  - keep the top contributing patches per page
  - pass both page and patch coordinates downstream

### 4. Hybrid Mode
- Integrate this with the existing Phase 5 pipeline in an optional hybrid configuration:
  - text + tables + image captions + visual patch retrieval
- This allows direct comparison against the current RAG pipeline

### Deliverables
- New retrieval module, e.g. `src/retrieval/late_interaction.py`
- Streamlit toggle for enabling Phase 6 visual retrieval

---

## 8. Phase 6.4: Contextual Generation with Visual Context

Once the relevant page patches are retrieved, the answer generation model must reason over the visual evidence directly.

### 1. Generation Context Construction
- Provide to the multimodal LLM:
  - the original user question
  - the top page image(s)
  - optionally cropped patches for focus
  - optional supporting text/chunk context from Phase 5

### 2. Answering Strategy
- Ask the multimodal LLM to:
  - read the visual evidence
  - identify the relevant region
  - extract exact figures when possible
  - synthesize the final answer with confidence and grounding

### 3. Prompt Design
- Prompts should instruct the model to:
  - prefer directly visible evidence on the page
  - mention uncertainty when the relevant data is not visible
  - avoid hallucinating unseen numeric values

### Deliverables
- Extend `src/generation/rag_chain.py` or create a Phase 6-specific multimodal answer chain
- Add support for passing page images / patch crops into Gemini

---

## 9. Phase 6.5: Enhanced Source Attribution

Phase 6 should make the system more transparent by showing not just the source page, but the exact visual region used.

### 1. Patch-Level Attribution
- Preserve the top-ranked patch coordinates through the entire pipeline
- Return:
  - page number
  - patch ID
  - bounding box coordinates
  - similarity score

### 2. UI Features
- In Streamlit, display:
  - source page image
  - highlighted patch/region
  - optional cropped evidence image

### 3. Answer Attribution
- Include a short source note such as:
  - “Source: Page 4, upper-center balance sheet region”

### Deliverables
- Streamlit visual source viewer for retrieved page patches
- Metadata handling for evidence overlays

---

## 10. Phase 6.6: Evaluation and Comparison with Previous Pipeline

This phase must be compared rigorously against the current Phase 5 system.

### 1. Evaluation Goals
- Measure whether direct visual retrieval improves:
  - table-heavy questions
  - chart interpretation
  - layout-sensitive questions
  - OCR-fragile or visually structured answers

### 2. Comparison Baselines
- **Baseline A:** Phase 5 text + table + image-caption RAG
- **Baseline B:** Phase 5 hybrid search + reranking
- **Baseline C:** Phase 6 visual-patch retrieval only
- **Baseline D:** Combined Phase 5 + Phase 6 hybrid pipeline

### 3. Metrics
- Reuse existing evaluation setup where possible:
  - RAGAS
  - LLM-as-a-Judge
- Add new multimodal-specific inspection:
  - whether the correct patch/page was retrieved
  - whether the cited patch overlaps the ground-truth evidence region

### 4. Manual Evaluation Categories
- Numeric extraction from tables
- Chart trend understanding
- Source localization accuracy
- Robustness to OCR/layout errors

### Deliverables
- Extend `evaluate.py`
- Add Phase 6 result slices to the evaluation dashboard

## 11. Recommended Implementation Breakdown

### Step 1: Page Rendering + Patch Extraction
- Build page-to-image pipeline
- Save page images and patch metadata

### Step 2: Visual Patch Indexing
- Choose initial multimodal embedding model
- Generate patch embeddings and store in Qdrant

### Step 3: Query-to-Patch Retrieval
- Implement late interaction / MaxSim scoring
- Return top patches and aggregate to pages

### Step 4: Multimodal Answer Generation
- Pass retrieved page images / patches to Gemini
- Generate answers grounded in visual evidence

### Step 5: Attribution UI
- Display page evidence and highlighted regions in Streamlit

### Step 6: Benchmark Against Phase 5
- Run side-by-side evaluation and analyze where visual retrieval actually helps

## 13. Risks and Challenges

- High storage cost for many patch embeddings
- Slower ingestion than Phase 5
- More complex retrieval scoring than standard cosine similarity
- Increased latency during generation if full page images are passed to the LLM
- Model selection trade-offs between quality, cost, and deployment complexity

---

## 14. Expected Outcome

After Phase 6, the system should be able to:
- retrieve answers from visually rich documents more directly,
- reason over tables/charts/layout without depending only on OCR or captions,
- provide stronger evidence grounding through page patch attribution,
- and be compared fairly against the current Phase 5 multimodal RAG pipeline.
